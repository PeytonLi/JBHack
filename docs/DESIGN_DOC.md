## Problem Statement

In production e-commerce systems, the gap between a crash being detected and a useful post-mortem being written is hours to days. Most of that time is human: engineers read Sentry, query the database, reproduce the bug, write the fix, open the PR, and then — often days later, if ever — write the Correction of Error (COE) document that tells the org what actually happened and how to prevent it.

AI tools today automate the code-fix half (Sentry AutoFix, Copilot Workspace, Aider, Devin, PR-Agent). Almost none generate the organizational COE artifact that Amazon-scale SRE culture treats as the real deliverable. Auto-Scribe closes that loop: Sentry webhook in, PR-with-COE out.

## What Makes This Cool

Three things that nobody else is stitching together:

1. **The COE as the finale, not a footnote.** Root cause, 5 whys, impact, prevention — rendered as a real Amazon-style post-mortem, not a git commit message.
2. **Live "thought log" dashboard.** The agent's reasoning streams into a web UI in real time (HTMX + SSE against the SQLite `agent_steps` table). Judges watching a video see the agent *think*, not a terminal scroll.
3. **Database-aware debugging.** The agent doesn't just read the stack trace — it generates a read-only SQL query against Supabase to see the exact row that crashed, then builds a pytest that injects that poison data.

## Recommended Approach

**Approach B, with a hard-cut fallback schedule.**

User explicitly chose the full spec despite a direct scope warning. Respect the conviction. But the 48-hour budget is real, so the execution plan below includes an **hour-40 cut line**: if the PR-creation step isn't working by then, fall back to writing the COE to disk and screenshotting it. The video doesn't need a real GitHub PR — it needs the *artifacts* of a real PR visible on screen.

Architecture (as written):

```
Sentry (real) ──▶ FastAPI /webhook/sentry ──▶ SQLite (agent memory)
                                              │
                                              ▼
                                      Claude Sonnet 4.6
                                      ├── parse stack trace
                                      ├── read codebase
                                      ├── read Supabase schema
                                      └── generate read-only SQL
                                              │
                                              ▼
                                      Claude generates pytest
                                              │
                                              ▼
                                      Docker sandbox (runs test, must fail)
                                              │
                                              ▼
                                      Claude generates code fix
                                              │
                                              ▼
                                      Docker sandbox (runs test, must pass)
                                              │
                                              ▼
                                      Claude Opus 4.7 writes COE_LOG.md
                                              │
                                              ▼
                                      GitHub PR (fix + test + COE)

                Live Dashboard (HTMX + SSE, polls SQLite agent_steps)
                watches all of the above in real time
```

Data stores:
- **Supabase (Postgres)**: `orders`, `inventory`, `warehouses`. Seeded with a known-broken row (a `warehouse_id` that doesn't exist in `warehouses`).
- **SQLite (local)**: `sessions`, `agent_steps`, `artifacts`. The agent's working memory and the dashboard's data source.

Models:
- `claude-sonnet-4-6` for stack trace parsing, SQL generation, pytest generation, fix generation.
- `claude-opus-4-7` for the COE writeup (the one place where reasoning quality shows up in the video).

The one canonical crash scenario (to avoid scope drift):
- `POST /checkout` on an order where `warehouse_id` is set to a value that doesn't exist in `warehouses`.
- The stack trace shows a `KeyError` or `TypeError` in a warehouse-resolution function.
- The fix: add a `.get()` fallback with a default warehouse, guarded by a log line.
- The COE root cause: missing foreign key constraint between `orders.warehouse_id` and `warehouses.id`.
- The prevention: add the constraint + add a Pydantic validator at the API boundary.

## Open Questions

1. **Sentry account setup time.** Is there a Sentry account ready with a project configured, or does this need to be spun up from zero? (Adds 30-60 min.)
2. **Supabase project setup.** Same question. RLS off for the agent's read-only queries, or role-based? (Recommend RLS off, single service-role key for hackathon.)
3. **GitHub PR target.** Real repo, or a throwaway repo created for the demo? (Recommend throwaway — you do not want the agent opening PRs against a real project during debugging.)
4. **Demo codebase.** Is "the Order Service" a real FastAPI repo you're also writing, or is there an existing repo the agent is debugging? If real, that's another scope item. Recommend a tiny single-file Flask/FastAPI service with ~3 endpoints as the "target system."

## Success Criteria

The 3-minute video demonstrates, end-to-end:

1. A crash happens on the target service (narrated: "customer tries to checkout").
2. Sentry fires the webhook. Dashboard lights up with a new session.
3. The thought log streams Claude's reasoning as it parses the trace, reads the schema, queries the row.
4. A pytest file is generated on screen, and it fails in the sandbox with the same error.
5. A fix is generated, the test re-runs in the sandbox and passes.
6. The COE_LOG.md is rendered in-video (Root Cause, Impact, Prevention, 5 Whys).
7. A GitHub PR is opened (or screenshotted, if falling back to the cut line).

If all 7 happen and the video is under 3 minutes, you win the category you're in.

## Scope Risk Callout

This plan is 56+ hours of work for a disciplined senior engineer. You have 48 solo hours. The expected outcome: you will finish 70-85% of this plan. The hard-cut schedule ensures that the 70% you finish is the 70% that matters for the video.

**If you hit any of these signals, cut harder:**
- Hour 20 and Sentry still isn't firing webhooks → fake the webhook payload, move on.
- Hour 28 and Docker sandbox is flaky → switch to `subprocess` with a fresh venv, move on.
- Hour 36 and you haven't written a single minute of video script → stop coding, script the video.

**Do not, under any circumstances:**
- Skip seed data and hope the agent works without a real poison row.
- Skip the COE step because "the fix works and that's what matters" — COE is the wedge.
- Start the dashboard at hour 40. It needs to exist, even skeletal, from hour 10 so you have the data hooks.
